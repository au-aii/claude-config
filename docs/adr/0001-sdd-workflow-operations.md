# 0001. SDD ワークフロー運用方針

## Context and Problem Statement

仕様駆動開発（SDD）ツールが 2025–2026 に乱立している（Kiro / GitHub Spec Kit /
Tessl / AWS AI-DLC）。「自前の SDD ワークフローを捨ててこれらに乗り換えるべきか、
乗り換えるならどれをどこまでか」が問い。

前提として、当環境には既に自前 SDD ワークフローが実装済み:
setup-project（永続ドキュメント群）→ asdd（4ステージ: ①計画[setup-steering:
requirements→design→tasklist / designer-agent・承認ゲート] ②批判検証[計画を疑う]
③実装[dev-agent] ④検証[implementation-validator 他]）→ doc-agent。
責務分離（design/dev/doc を別エージェント）・承認ゲート・小変更エスケープハッチを備える。

成熟度は Level 3.5 / 5（CMM 類推）。Defined(L3) は達成しているが、一貫性に2つの穴があり
Managed(L4) に未達:

- **穴A（入口の重複）**: 検証段階に implementation-validator / code-reviewer / doc-reviewer /
  reviewer-agent が併存し、「どれを呼ぶか」が一意でない。
- **穴B（配置の不一致）**: spec / .steering の正本配置がリポジトリ間で揺れ、同じ手順が
  環境ごとに別物になる。

## Decision Drivers

- **一貫性**: 同じワークフローがどのリポジトリでも一意に動くこと
- **最良性**: requirements の品質・整合性チェックの強化
- **乱立・混在の回避**: ツールやファイルの総数を増やしたくない
- **ロックイン回避**: SDD ツールは発展途上で、早すぎる固定化は負債になる

## Considered Options

1. Kiro へ乗り換え（AWS の独立 AI IDE。requirements/design/tasks ＋ Agent Hooks ＋ Steering）
2. GitHub Spec Kit を導入（OSS・エージェント非依存 CLI。/speckit.\* コマンド群）
3. Tessl へ乗る（仕様ファースト ＋ spec registry）
4. AWS AI-DLC を採用（方法論）
5. 自前ワークフローを維持し、外部の良いパターンだけ吸収する

調査所見（2026-06 時点）:

- 4ツール共通の段階構成は requirements→design→tasks→implement で、自前 setup-steering と同型。
- **Kiro**: 独立 IDE・ロックイン中〜高・プレビュー→GA 移行期。
- **GitHub Spec Kit**: OSS・Claude Code 併用可・ロックイン低・ただし実験的（初期段階）。
- **Tessl**: 未成熟・招待制・独自エコシステム。
- **AWS AI-DLC**: ツールではなく方法論・概念段階。
- 自前の強み: 批判検証ステージ（計画を疑う）を内蔵（外部ツールにない）。責務分離が徹底。
- 自前の弱み: EARS 記法未採用 / 整合性チェックが弱い / spec の再利用・バージョン管理なし /
  テスト連携が「実行して確認」止まり / spec・.steering の置き場所が曖昧。

## Decision Outcome

**選択: Option 5（自前維持 ＋ パターン吸収）**

理由: 4ツール共通の段階構成は自前と同型で、乗り換えの本質的な新規性が乏しい。むしろ
自前は批判検証という優位を持つ。外部ツールへの乗り換えはロックイン・未成熟リスク・自前資産の
放棄を伴う。よって乗り換えず、外部からは「記法・段階」というパターンのみを自前に溶かす。
かつ、最良性（EARS 等の導入）より先に一貫性（2つの穴）を塞ぐ — 順序を守らないと、一貫性の
ない場所に高品質な要件を載せることになり効果が出ないため。

### 導入の4規律（混在・乱立の防止線）

1. **正本は asdd 一本**（外部ツールは常駐させない。Spec Kit も「一度触って観察」のみ）
2. **借りるのは「記法・段階」であり「ツール・コマンド」ではない**（自前に溶かし、新スキル/依存を作らない）
3. **純増禁止**（1つ足すなら1つ棚卸し。穴A の重複削除を先に）
4. **置き場所は1つに固定**（spec / .steering の正本を所有権ルールで一意化）

### ロードマップ

| ステップ            | スコープ                                                                | 目的                 |
| ------------------- | ----------------------------------------------------------------------- | -------------------- |
| **Step0**（最優先） | 穴A: 検証入口 4→1 統合 / 穴B: spec・.steering 配置の一意化              | 一貫性の確保         |
| **Step1**           | EARS 記法を requirements に溶かす / 整合性チェックを asdd Stage2 に追加 | 最良性の向上         |
| **Step2**（保留）   | spec→テスト自動連携 / spec registry                                     | 型が固まってから判断 |

### Consequences

**良い点:**

- 成熟度 L3.5→L4。経路と配置が一意になりワークフローが完成に近づく。
- ツール総数を増やさず乱立・混在を構造的に回避。
- 業界標準（EARS・整合性チェック）の良い部分だけ低コストで吸収。

**悪い点:**

- 自前ゆえ外部ツールのアップデート恩恵は受けられない（メンテは自己負担）。
- spec の再利用・registry は当面持たない（意図的な保留）。
